Codex Cookbook

让团队成员能快速抄走
稳定复用

不是堆提示词，而是把常见协作动作收成一套能直接发给 Codex 的任务卡。

核心目标 少试错

把目标、边界、输出、验证一次说清。

适用场景 日常开发

诊断、实现、复盘、复查、项目收敛。

01

先记住 3 件事

  1. 不要只提目标，还要说明工作方式、范围和输出格式。
  2. 任务明确就直接推进，边界不清再先收敛。
  3. 不要只让它“做”，还要让它“证明做对”。
一句话总结

Codex 更像执行搭档，不是读心机器。

01

一个统一骨架


目标：解决【具体问题】；如果适合，尽量逼近【最终可用状态】。
模式：【直接推进 / 先诊断 / 先复现 / 先收敛计划】。
约束：这次只解决【范围 A】，不扩展到【范围 B】；优先【长期可维护 / 短期可交付】。
输出：先给【判断 / 方案 / 完整计划】，再【落地 / 继续问关键问题】。
验证：通过【测试 / 最小验证步骤 / 验收标准】证明结果有效。

这些模板默认都是写给 Codex 的，可以直接复制后替换占位内容。

01

3 句高频补充

这次只解决当前核心问题，不顺手扩展、不做无关重构。
如果有需要我确认的问题，请只提最关键的 1 到 3 个。
先列出关键假设和缺失信息；能自行拍板的直接继续。
02

八个高频场景

01 先诊断

先找问题，再决定怎么改。

02 直接推进

任务明确时别机械讨论。

03 只问关键问题

把低价值来回确认降到最低。

04 长期可维护

把短期解和长期解讲清楚。

05 失败先复盘

先判断错在目标、路径还是执行。

06 补验证闭环

不要停在“代码已改”。

07 先复现再修

修 bug 不靠盲改。

08 先收敛计划

复杂任务先查漏补缺。

03

直接开发型


目标：解决【具体问题】，并尽量逼近【最终可用方案】。
模式：如果任务已经明确，就直接推进。
约束：先读上下文再动手；非关键问题不要频繁停下来；
      这次只解决【当前核心问题】，不顺手扩展到【无关范围】。
输出：先给问题判断和推荐方案，再直接落地；
      如果有必须我拍板的问题，按“问题、选项、利弊、推荐”来问我。
验证：改完后补充【测试 / 最小验证闭环】。
03

Bug 修复型


目标：修复【某个 bug】，并避免盲改。
模式：一般不需要先做计划收敛。
约束：优先先复现，再修复。
输出：说明 bug 根因、如何复现、代码如何修复。
验证：确认问题先被稳定复现，再确认修复后通过。
关键差别

不是让它直接改，而是先把问题钉死，再让修复有依据。

03

独立复查型

  • 适合主流程基本完成后使用。
  • 更适合复查、补充视角和并行审视。
  • 不适合承接必须立刻解决的主线任务。
复查维度

从【前端 / 架构 / 接口 / 文档】视角，检查【稳定性 / 可维护性 / 一致性】。

04

中型项目怎么用

01

先对齐业务目标、场景边界和不做项。

02

文档未定时，先做计划收敛，把关键共识定下来。

03

文档基本定下来后，再用计划模式查漏补缺，整理出完整计划。

04

进入执行阶段后，默认直接推进，不常驻计划模式。

04

这一步最关键


文档基本定下来后，先进入计划模式，
帮我补扫一遍我可能没想到的问题和遗漏约束；
然后基于当前共识，直接整理成可推进开发的完整执行计划。

这不是“再讨论一轮”，而是“查漏补缺后直接开工”。

05

常见反模式

只写 review

太泛。要写清楚是代码、UI、架构还是文档 review。

只写 test

要区分写测试、跑测试和人工验证。

一直问还有什么要确认

会把低价值问题重新抛回来。

不写范围

容易顺手扩展，最后越改越多。

机械先做计划

简单任务也被拉成长链路。

06

最后只记这 5 句

目标：要解决什么问题。
模式：直接推进，还是先诊断、先复现、先收敛。
约束：范围到哪里为止，优先长期还是短期。
输出：要方案、代码、文档、测试，还是完整执行计划。
验证：怎么证明这次结果是对的。
