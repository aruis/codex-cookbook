# Codex Cookbook

面向公司内部分享的精简版手册。目标只有一个：让团队成员能快速抄走、稳定复用。

## 先记住 3 件事

1. 不要只提目标，还要说明工作方式、范围和输出格式。
2. 不要默认所有任务都先讨论；任务明确就直接推进，边界不清再先收敛。
3. 不要只让它“做”，还要让它“证明做对”。

## 一个统一骨架

下面这些模板默认都是写给 Codex 的。可以直接复制，再按任务替换占位内容。

写 Codex 提示词时，优先按这个顺序表达：

1. 目标：要解决什么问题。
2. 模式：直接推进，还是先收敛计划、先诊断、先复现。
3. 约束：范围边界、质量取向、时间取舍。
4. 输出：要方案、代码、文档、测试、复盘，还是完整执行计划。
5. 验证：怎么证明结果是对的。

推荐骨架：

```text
目标：解决【具体问题】；如果适合，尽量逼近【最终可用状态】。
模式：【直接推进 / 先诊断 / 先复现 / 先收敛计划】。
约束：这次只解决【范围 A】，不扩展到【范围 B】；优先【长期可维护 / 短期可交付】。
输出：先给【判断 / 方案 / 完整计划】，再【落地 / 继续问关键问题】。
验证：通过【测试 / 最小验证步骤 / 验收标准】证明结果有效。
```

常用补充句：

```text
这次只解决当前核心问题，不顺手扩展、不做无关重构。

如果有需要我确认的问题，请只提最关键的 1 到 3 个，并按“问题、选项、利弊、推荐”格式来问。

先列出你的关键假设和缺失信息；能自行拍板的直接继续，不能拍板的最后集中列出。
```

说明：

- `review` 不等于代码 review。最好写清楚是代码、UI、架构、文档，还是业务方案 review。
- “先补失败测试”适合大多数 bug，但不是所有问题都适合自动化测试；不适合时，就改成最小可行验证步骤。
- 文中提到的“计划模式”“subagent”只适用于支持这些能力的环境。

## 八个高频场景

### 1. 先诊断，不急着改

适用：

- 想看 UI、代码、方案哪里还不够好
- 任务还比较模糊
- 想先找问题，再决定是否开工

模板：

```text
先不要急着改。先从【指定维度】系统指出当前存在的问题，并按优先级排序；如果需要，再补充你推荐的修正方向。
```

误用边界：

- 不限定“指定维度”，输出容易发散。
- 任务已经很明确时，没必要先绕回诊断。

### 2. 任务明确，直接推进

适用：

- 方向已经清楚
- 不想频繁来回确认
- 更关心结果，不想被过程噪音打断

模板：

```text
你直接按逼近最终目标的思路推进，先读上下文再动手；非关键问题不要频繁停下来，只有你不能拍板的问题再集中问我。
```

误用边界：

- 边界不清时，容易顺手扩散改动，所以最好同时写上范围约束。

### 3. 只把关键问题抛给我

适用：

- 只想参与关键决策
- 不希望被低质量问题打断
- 希望它先自行收敛，再把真正需要拍板的点拿出来

模板：

```text
如果有需要我确认的问题，请只提最关键的 1 到 3 个，并按“问题、选项、利弊、推荐”格式来问。
```

误用边界：

- 如果你一句“有问题就问”都不加，它很可能把低价值问题也抛回来。

### 4. 优先长期可维护

适用：

- 这次改动后面还会继续演进
- 你担心短期通过掩盖长期问题

模板：

```text
不要为了短期通过牺牲长期可维护性、健壮性和扩展性。优先选择中长期站得住的方案；如果必须做短期方案，请明确标注取舍和后续风险。
```

误用边界：

- 如果业务本身就是一次性或临时方案，要明确告诉它可以接受短期解。

### 5. 连续失败后，先复盘再继续

适用：

- 已经反复试了多轮
- 怀疑方向本身不对
- 需要重新对齐目标、路径和停点

模板：

```text
这件事已经反复失败多次。请先判断是目标错、路径错，还是执行细节错，并给出下一轮最值得尝试的方案，以及建议停掉哪些低收益动作。
```

误用边界：

- 如果不要求“下一轮方案”，很容易停留在空泛复盘。

### 6. 做完以后补验证闭环

适用：

- 不想只停留在“代码已改”
- 希望它证明这次改动真的有效
- 希望它先给出验收标准，再实现和验证

模板：

```text
改完后不要停。先写出这次完成的验收标准，再补充必要测试或验证步骤，证明这次修改确实解决了问题，没有靠硬压、绕过或碰巧通过。
```

误用边界：

- 只写 `test` 太模糊，容易得到低质量结果。

### 7. 先复现，再修复

适用：

- 正在修 bug
- 希望先锁定问题，再改代码
- 不希望修复过程变成盲改

模板：

```text
先不要直接改代码。优先补一个最小失败测试来稳定复现这个 bug；如果不适合写自动化测试，就先给出最小可行复现和验证步骤。确认问题被稳定复现后，再修改代码并检查没有引入误伤。
```

误用边界：

- 不要为了让测试通过而把测试写得过宽或偏离真实 bug。
- 测试要尽量最小化，只锁定这次问题，不要顺手把无关行为都绑进去。

### 8. 任务复杂时，先收敛计划

适用：

- 任务目标还不够清楚
- 需求里有多个分支或决策点
- 你希望先讨论方案、拆步骤、确认边界，再进入实现

模板：

```text
先不要急着落代码。先把目标、约束、范围边界、关键分歧点、推荐路径和执行步骤梳理清楚；如果有需要我确认的地方，按优先级逐条拿出来。
```

误用边界：

- 任务已经很直接时，不要机械开启计划模式，否则会拖慢推进。
- 计划模式适合收敛问题和确认路径，不适合代替实际落地。

## 四个推荐组合

### 1. 方案讨论型

```text
目标：先收敛【某个方案 / 某个模块怎么做】，先不急着写代码。
模式：必要时先做计划收敛。
约束：把真正关键的分歧点提炼出来，不要把低价值问题一起抛给我。
输出：按“问题、选项、利弊、推荐”格式整理，我来拍板。
验证：明确这次准备解决什么、不解决什么，以及为什么。
```

### 2. 直接开发型

```text
目标：解决【具体问题】，并尽量逼近【最终可用方案】。
模式：如果任务已经明确，就直接推进。
约束：先读上下文再动手；非关键问题不要频繁停下来；这次只解决【当前核心问题】，不顺手扩展到【无关范围】。
输出：先给问题判断和推荐方案，再直接落地；如果有必须我拍板的问题，按“问题、选项、利弊、推荐”来问我。
验证：改完后补充【测试 / 最小验证闭环】。
```

### 3. Bug 修复型

```text
目标：修复【某个 bug】，并避免盲改。
模式：一般不需要先做计划收敛，除非背后牵涉较大范围的方案调整。
约束：优先先复现，再修复；不要为了让测试通过而绕开真实问题。
输出：说明 bug 根因、如何复现、代码如何修复。
验证：确认问题先被稳定复现，再确认修复后通过，并补充必要的回归验证。
```

### 4. 独立复查型

```text
目标：对当前结果做一轮独立复查，而不是重复主流程。
模式：适合在主流程基本完成后使用。
约束：从【前端 / 架构 / 接口 / 文档】视角重点检查【稳定性 / 可维护性 / 一致性】；聚焦复查，不重复实现。
输出：指出风险、遗漏、更优建议，以及哪些点已经足够，不需要继续投入。
验证：把“必须改”和“可选优化”分开列出。
```

说明：

- subagent 更适合复查、补充视角和并行审视，不适合承接必须立刻解决的主线任务。

## 中型项目推荐工作流

适用：

- 在做中型项目或独立模块
- 业务和场景需要先对齐，再进入实现
- 你希望先把文档、边界和开发路径收敛清楚

推荐流程：

1. 先对齐业务目标、场景边界和不做项，不急着落代码。
2. 文档未定时，先做计划收敛，把关键共识定下来。
3. 文档基本定下来后，再用计划模式查漏补缺，整理出可直接推进的完整计划。
4. 进入执行阶段后，默认直接推进，不要把计划模式常驻开启。
5. 只有在阶段切换、明显偏航、反复失败或需要重新收敛时，再回到计划模式。
6. 关键阶段完成后，再做一次独立复查。

推荐模板：

```text
这是一个中型项目 / 模块。先不要直接开始实现。
第一步，先把业务目标、场景边界、约束和不做项收敛清楚。
第二步，文档基本定下来后，再用计划模式查漏补缺，整理成可直接推进的完整计划。
第三步，进入执行阶段后默认直接推进；只有在阶段切换、偏航或卡住时，再回到计划收敛。
第四步，关键阶段结束后，再补一轮独立复查。
```

## 常见反模式

### 1. 只写 `review`

问题：

- 太泛了
- 容易只做表面检查

更好的写法：

```text
按【代码 / UI / 架构 / 文档 / 业务方案】review 的方式复查，优先找出最重要的问题，并按优先级排序。
```

### 2. 只写 `test`

问题：

- 不知道到底是写测试、跑测试，还是人工验证

更好的写法：

```text
如果是在修 bug，优先补一个最小失败测试稳定复现问题，再修改代码并确认测试通过；如果不适合写自动化测试，就给出最小可行的验证步骤和结果。
```

### 3. 一直问“还有什么要确认的么”

问题：

- 容易把大量低价值问题重新抛回来

更好的写法：

```text
如果有需要我确认的问题，请只提关键问题，并带上你的推荐方案和理由。
```

### 4. 不写范围，导致越改越多

问题：

- 容易顺手扩展和重构

更好的写法：

```text
这次只解决当前核心问题，不顺手扩展、不做无关重构；如果发现更大问题，先记录并告诉我，不要直接扩散改动。
```

### 5. 任务不复杂也机械先做计划

问题：

- 会把本来能直接推进的任务拉长
- 容易从“解决问题”滑到“讨论问题”

更好的写法：

```text
如果任务边界已经明确，就直接推进；只有在目标、路径或边界不清时，再先做计划收敛。
```

## 一页版随手抄

```text
目标：要解决什么问题。
模式：直接推进，还是先诊断、先复现、先收敛。
约束：范围到哪里为止，优先长期还是短期。
输出：要方案、代码、文档、测试，还是完整执行计划。
验证：怎么证明这次结果是对的。

这次只解决当前核心问题，不顺手扩展、不做无关重构。

如果有需要我确认的问题，请只提最关键的 1 到 3 个，并带上推荐方案和理由。

改完后不要停，请补上测试或最小验证闭环。

修 bug 时，优先先复现，再修复；不适合写自动化测试时，就给出最小可行验证步骤。

任务复杂、边界不清时，先做计划收敛；任务明确时不要机械开启。
```
