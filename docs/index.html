道

Dao

先讲底层判断

这一章只回答一件事：为什么有些协作会失焦、失真、失稳。

道

三个底层判断，决定结果上限

01

以边界换聚焦

不给边界，Codex 就只能平均用力；明确不做什么、什么可妥协、什么不能妥协，结果才会真正聚焦。

02

以信息换准确

你提供的信息密度，决定结果和预期的贴合度；目标、背景、约束和细节说得越清楚，返工越少。

03

以文档和验证换稳定

上下文会衰减，口头共识会丢失；目标、方案、进度要尽早落文档并持续更新，结果还要靠测试或验证闭环来证明。

法

Fa

再讲协作如何组织

“法”解决协作组织。目标、边界、阶段、验证，要被组织成一条可执行的链。

法

常见任务里，先按这 6 条组织配合

01

先收敛，再展开。目标、边界、路径不清时，先收敛问题。

02

明确任务的时间尺度。长期建设、阶段性过渡、短期止血，是三种不同任务。

03

根据阶段定模式。方案期适合讨论和计划，问题模糊时适合先诊断。

04

用结构化表达降低偏差，让目标、约束、输出和验证都清楚可读。

05

用文档沉淀阶段共识。多轮协作不要只靠会话记忆。

06

用验证闭环约束结果。不要只停在“已经改了”。

法

单次任务里，这 6 条通常会压缩成一个最小闭环

目标 模式 约束 输出 验证

它的价值不在格式本身，而在于提前说清偏差最容易出现的地方。

法

推荐骨架

目标：解决【具体问题】；如果适合，尽量逼近【最终可用状态】。

模式：【直接推进 / 先诊断 / 先复现 / 先收敛计划】。

约束：这次只解决【范围 A】，不扩展到【范围 B】；这次任务属于【长期建设 / 阶段性过渡 / 短期止血】，优先【长期可维护 / 短期可交付】。

输出：先给【判断 / 方案 / 完整计划】，再【落地 / 继续问关键问题】。

验证：通过【测试 / 最小验证步骤 / 验收标准】证明结果有效。

术 · 第一组

先判断任务清晰度，再决定是诊断、计划还是直接推进

1. 局面未明：先诊断，不急着改

适用：想先找问题，再决定是否开工；任务还比较模糊。

模板：先不要急着改。先从【指定维度】系统指出当前存在的问题，并按优先级排序；如果需要，再补充你推荐的修正方向。

误用边界：不限定“指定维度”，输出容易发散。

2. 主线清晰：直接推进

适用：方向已经清楚，不想频繁来回确认，更关心结果。

模板：你直接按逼近最终目标的思路推进，先读上下文再动手；非关键问题不要频繁停下来，只有你不能拍板的问题再集中问我。

误用边界：边界不清时容易顺手扩散改动，所以最好同时写上范围约束。

10. 任务复杂：先收敛计划

适用：目标不够清楚，需求里有多个分支或决策点。

模板：先不要急着落代码。先把目标、约束、范围边界、关键分歧点、推荐路径和执行步骤梳理清楚；如果有需要我确认的地方，按优先级逐条拿出来。

误用边界：任务已经很直接时，不要机械开启计划模式。

术 · 第二组

时间尺度不同，打法就不同

3. 目标偏长期建设：优先长期可维护

适用：这次改动后面还会继续演进，担心短期通过掩盖长期问题。

模板：不要为了短期通过牺牲长期可维护性、健壮性和扩展性。优先选择中长期站得住的方案；如果必须做短期方案，请明确标注取舍和后续风险。

4. 目标偏短期止血：优先恢复可用

适用：线上问题需要先恢复，当前目标是快速止损，而不是一次性彻底治理。

模板：这次以短期止血为主，优先恢复可用和控制影响面，不追求一次性彻底治理；如果用了临时方案，请标注后续治理项。

5. 不需要兼容过渡：直接硬切

适用：明确知道不需要兼容层、双写、灰度适配或过渡方案。

模板：这次不考虑兼容过渡，直接按目标状态修改；不要额外保留兼容层、桥接逻辑或过渡方案。如果会破坏存量契约，请直接同步调整调用方和相关实现。

术 · 第三组

临近交付时，重点是复现、验证和避免盲改

7. 临近收口：补验证闭环

适用：不想只停留在“代码已改”，希望证明这次改动真的有效。

模板：写出这次完成的验收标准，再补充必要测试或验证步骤，证明这次修改确实解决了问题，而不是靠硬压、绕过或碰巧通过。

8. Bug 场景：先复现，再修复

适用：正在修 bug，希望先锁定问题，再改代码。

模板：先不要直接改代码。优先补一个最小失败测试来稳定复现这个 bug；如果不适合写自动化测试，就先给出最小可行复现和验证步骤。确认问题被稳定复现后，再修改代码并检查没有引入误伤。

术 · 第四组

任务跨多轮或反复失败时，要切换到复盘与阶段推进

6. 连续失败后：先复盘，再继续

模板：这件事已经反复失败多次。请先判断是目标错、路径错，还是执行细节错，并给出下一轮最值得尝试的方案，以及建议停掉哪些低收益动作。

9. 多轮协作：先落文档，再继续推进

模板：这件事会跨多轮推进。请先把当前目标、关键约束、已达成结论、待决问题和下一步计划整理成文档或结构化记录；后续推进过程中持续更新，不要只依赖会话上下文。

11. 中型项目：按阶段推进

推荐流程：先对齐业务目标、场景边界和不做项；文档未定时先做计划收敛；执行阶段默认直接推进；只有在阶段切换、偏航或卡住时，再回到计划收敛；关键阶段结束后，再补一轮独立复查。

器

Qi

最后才进入拿来即用

这一章不再讲判断，而是直接给能复制的模板、反模式和补充句。

器

四个推荐组合

方案讨论型

目标：先收敛【某个方案 / 某个模块怎么做】，先不急着写代码。

模式：必要时先做计划收敛。

输出：按“问题、选项、利弊、推荐”格式整理，我来拍板。

直接开发型

目标：解决【具体问题】，并尽量逼近【最终可用方案】。

约束：先读上下文再动手；非关键问题不要频繁停下来；只解决当前核心问题。

验证：改完后补充【测试 / 最小验证闭环】。

Bug 修复型

目标：修复【某个 bug】，并避免盲改。

约束：优先先复现，再修复；不要为了让测试通过而绕开真实问题。

输出：说明 bug 根因、如何复现、代码如何修复。

独立复查型

目标：对当前结果做一轮独立复查，而不是重复主流程。

约束：从【前端 / 架构 / 接口 / 文档】视角重点检查【稳定性 / 可维护性 / 一致性】。

验证：把“必须改”和“可选优化”分开列出。

器

常见反模式，不是不能写，而是太容易写得失真

只写 review

问题：太泛了，容易只做表面检查。

更好的写法：按【代码 / UI / 架构 / 文档 / 业务方案】review 的方式复查，优先找出最重要的问题，并按优先级排序。

只写 test

问题：不知道到底是写测试、跑测试，还是人工验证。

更好的写法：如果是在修 bug，优先补一个最小失败测试稳定复现问题，再修改代码并确认测试通过；如果不适合写自动化测试，就给出最小可行的验证步骤和结果。

一直问“还有什么要确认的么”

问题：容易把大量低价值问题重新抛回来。

更好的写法：如果有需要我确认的问题，请只提关键问题，并带上你的推荐方案和理由。

不写范围，导致越改越多

问题：容易顺手扩展和重构。

更好的写法：这次只解决当前核心问题，不顺手扩展、不做无关重构；如果发现更大问题，先记录并告诉我，不要直接扩散改动。

器

通用补充句，可以直接复用在不同任务里

这次只解决当前核心问题，不顺手扩展、不做无关重构。

这次不考虑兼容过渡，直接按目标状态修改；不要额外保留兼容层、桥接逻辑或过渡方案。

如果有需要我确认的问题，请只提最关键的 1 到 3 个，并按“问题、选项、利弊、推荐”格式来问。

先列出你的关键假设和缺失信息；能自行拍板的直接继续，不能拍板的最后集中列出。

收束一句话：道定方向，法定组织，术定打法，器定复用。

用结构降低偏差，用验证确保结果。
