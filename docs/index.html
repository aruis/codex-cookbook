Dao

先讲底层判断

这一章只回答一件事：为什么有些协作会失焦、失真、失稳。

道

三个底层判断，决定结果上限

以边界换聚焦

不给边界，Codex 就只能平均用力；明确不做什么、什么可妥协、什么不能妥协，结果才会真正聚焦。

以信息换准确

你提供的信息密度，决定结果和预期的贴合度；目标、背景、约束和细节说得越清楚，返工越少。

以文档和验证换稳定

上下文会衰减，口头共识会丢失；目标、方案、进度要尽早落文档并持续更新，结果还要靠测试或验证闭环来证明。

Fa

再讲协作如何组织

“法”解决协作组织。目标、边界、阶段、验证，要被组织成一条可执行的链。

法

目标、边界、路径不清时，先收敛问题。

长期建设、阶段性过渡、短期止血，是三种不同任务，方案选择标准也不同。

方案期适合讨论和计划，问题模糊时适合先诊断，修 bug 适合先复现，主线清晰后再进入执行，阶段结束后再独立复查。

不要想到什么说什么，尽量把请求组织成稳定结构，让目标、约束、输出和验证标准都清楚可读。

多轮协作不要只靠会话记忆，目标、方案、决策、进度和风险要尽早外化，并持续更新。

不要只停在“已经改了”，要用测试、验收标准或最小验证步骤确认结果真的成立。

准备采用临时解前，先说明更优路径为什么暂时不可行；实施前先确认，实施后再验证并说明风险与回收方式。

法

在单次任务里，这 7 条通常可以压缩成一个最小闭环。

目标 模式 约束 输出 验证

它的价值不在格式本身，而在于提前说清偏差最容易出现的地方。

法

推荐骨架

目标：解决【具体问题】；如果适合，尽量逼近【最终可用状态】。

模式：【直接推进 / 先诊断 / 先复现 / 先收敛计划】。

约束：这次只解决【范围 A】，不扩展到【范围 B】；这次任务属于【长期建设 / 阶段性过渡 / 短期止血】，优先【长期可维护 / 短期可交付】。

输出：先给【判断 / 方案 / 完整计划】，再【落地 / 继续问关键问题】。

验证：通过【测试 / 最小验证步骤 / 验收标准】证明结果有效。

Shu

进入具体局面后，再选择打法

这一层解决的是“进入具体局面后，应该怎么出手”。下面不是零散技巧，而是几类高频局面下的常用打法。

术 · 01

局面未明：先诊断，不急着改

适用

  • 想看 UI、代码、方案哪里还不够好
  • 任务还比较模糊
  • 想先找问题，再决定是否开工
  • 你懒得一开始就把背景讲得很细
  • 你只是想先看看 AI 能给出什么建议

模板

先不要急着改。先从【指定维度】系统指出当前存在的问题，并按优先级排序；如果需要，再补充你推荐的修正方向。

如果还不想讲太细，也可以先用引导性提问开局，比如：先基于【当前现状 / 目标方向】，给我 3 到 5 条值得优先考虑的建议，并说明各自适用的前提。先从【产品 / 代码 / 方案 / 体验】这些角度看，指出当前最值得注意的问题，以及为什么它们值得优先关注。

误用边界

  • 不限定“指定维度”，输出容易发散。
  • 任务已经很明确时，没必要先绕回诊断。
  • 真要进入落地时，还是要把目标、约束和范围补齐。
  • 最好要求它先列出关键假设、缺失信息，并区分哪些判断基于已知事实、哪些只是启发式建议。
  • 如果输出过泛，先让它指出还缺哪 1 到 3 个最关键的信息，再继续收敛。

术 · 02

主线清晰：直接推进

适用

  • 方向已经清楚
  • 不想频繁来回确认
  • 更关心结果，不想被过程噪音打断

模板

你直接按逼近最终目标的思路推进，先读上下文再动手；非关键问题不要频繁停下来，只有你不能拍板的问题再集中问我。

误用边界

  • 边界不清时，容易顺手扩散改动，所以最好同时写上范围约束。

术 · 03

目标偏长期建设：优先长期可维护

适用

  • 这次改动后面还会继续演进
  • 你担心短期通过掩盖长期问题

模板

不要为了短期通过牺牲长期可维护性、健壮性和扩展性。优先选择中长期站得住的方案；如果必须做短期方案，请明确标注取舍和后续风险。

误用边界

  • 如果业务本身就是一次性或临时方案，要明确告诉它可以接受短期解。

术 · 04

目标偏短期止血：优先恢复可用

适用

  • 线上问题需要先恢复
  • 当前目标是快速止损，而不是一次性彻底治理
  • 你可以接受阶段性技术债，但不能接受继续失效

模板

这次以短期止血为主，优先恢复可用和控制影响面，不追求一次性彻底治理；如果用了临时方案，请标注后续治理项。

误用边界

  • 如果当前不是应急场景，就不要用止血思路替代长期建设。
  • 即使是止血，也要说明遗留风险和后续治理事项。

术 · 05

不需要兼容过渡：直接硬切

适用

  • 这次改动会破坏存量契约
  • 你明确知道不需要兼容层、双写、灰度适配或过渡方案
  • 你希望直接切到目标状态

模板

这次不考虑兼容过渡，直接按目标状态修改；不要额外保留兼容层、桥接逻辑或过渡方案。如果会破坏存量契约，请直接同步调整调用方和相关实现。

误用边界

  • 如果真实场景仍然需要平滑迁移、灰度发布或跨版本兼容，就不要强行硬切。
  • 决定硬切前，要确认受影响范围和调用方是可控的。

术 · 06

连续失败后：先复盘，再继续

如果问题已经连续失败、或者你怀疑方向本身不对，用这一节；如果不是失败，而是链路越走越深、开始怀疑继续投入是否还划算，用下一节。

适用

  • 已经反复试了多轮
  • 怀疑方向本身不对
  • 需要重新对齐目标、路径和停点

模板

这件事已经反复失败多次。请先判断是目标错、路径错，还是执行细节错，并给出下一轮最值得尝试的方案，以及建议停掉哪些低收益动作。

误用边界

  • 如果不要求“下一轮方案”，很容易停留在空泛复盘。

术 · 07

链路越走越深：先评估停点，再决定是否继续

如果问题没有彻底失败，但你开始怀疑继续深挖的投入产出比是否还成立，用这一节；如果已经连续失败、或者怀疑方向本身不对，优先回到上一节先复盘。

适用

  • 一条分支链路已经展开了多轮
  • 还没彻底失败，但收益开始变得不确定
  • 你怀疑继续深挖的投入产出比已经下降
  • 你想判断现在更适合继续、收口、换路还是暂停

模板

先不要继续顺着当前链路往下做。先评估一下这条链路到目前为止的投入产出比：已经获得了什么、还缺什么、继续投入最可能带来什么收益、主要风险和成本是什么。然后判断现在更适合继续深挖、先收口、切换方向，还是在当前停住，并说明理由。

误用边界

  • 不要每推进一步都机械做停点评估，否则会打断正常执行。
  • 这更适合链路已经明显拉长、或者你开始怀疑继续投入是否还划算的时候。
  • 重点不是复述过程，而是判断“现在最合适的动作”。
  • 如果目标本身还清楚、链路也还在稳定产出，就没必要过早收口。

术 · 08

临近收口：补验证闭环

适用

  • 不想只停留在“代码已改”
  • 希望它证明这次改动真的有效
  • 希望它先给出验收标准，再实现和验证

模板

写出这次完成的验收标准，再补充必要测试或验证步骤，证明这次修改确实解决了问题，而不是靠硬压、绕过或碰巧通过。

误用边界

  • 不要把验收标准写成空话，验证步骤要能真正证明问题已解决。
  • 不要为了补验证顺手扩大改动范围。

术 · 09

Bug 场景：先复现，再修复

适用

  • 正在修 bug
  • 希望先锁定问题，再改代码
  • 不希望修复过程变成盲改

模板

先不要直接改代码。优先补一个最小失败测试来稳定复现这个 bug；如果不适合写自动化测试，就先给出最小可行复现和验证步骤。确认问题被稳定复现后，再修改代码并检查没有引入误伤。

误用边界

  • 不要为了让测试通过而把测试写得过宽或偏离真实 bug。
  • 测试要尽量最小化，只锁定这次问题。

术 · 10

多轮协作：先落文档，再继续推进

适用

  • 任务会跨多轮对话持续推进
  • 目标、方案、进度已经开始变复杂
  • 你担心上下文压缩后丢失关键共识

模板

这件事会跨多轮推进。请先把当前目标、关键约束、已达成结论、待决问题和下一步计划整理成文档或结构化记录；后续推进过程中持续更新，不要只依赖会话上下文。

误用边界

  • 很小的一次性任务，不需要为了形式而额外维护文档。
  • 文档不是一次性快照，后续关键变化要同步更新。

术 · 11

任务复杂：先收敛计划（推荐开启计划模式）

适用

  • 任务目标还不够清楚
  • 需求里有多个分支或决策点
  • 你希望先讨论方案、拆步骤、确认边界，再进入实现

模板

先不要急着落代码。先把目标、约束、范围边界、关键分歧点、推荐路径和执行步骤梳理清楚；如果有需要我确认的地方，按优先级逐条拿出来。

误用边界

  • 任务已经很直接时，不要机械开启计划模式，否则会拖慢推进。
  • 计划模式适合收敛问题和确认路径，不适合代替实际落地。

术 · 12

中型项目：按阶段推进

适用

  • 在做中型项目或独立模块
  • 业务和场景需要先对齐，再进入实现
  • 你希望先把文档、边界和开发路径收敛清楚

推荐流程

  1. 先对齐业务目标、场景边界和不做项，不急着落代码。
  2. 文档未定时，先做计划收敛，把关键共识定下来。
  3. 文档基本定下来后，再用计划模式查漏补缺，整理出可直接推进的完整计划。
  4. 如果方案刚收敛完、你又担心有遗漏，可以先加一个可选复核步骤，让辅助代理或独立视角先审一遍方案，再决定是否进入执行。
  5. 进入执行阶段后，默认直接推进，不要把计划模式常驻开启。
  6. 只有在阶段切换、明显偏航、反复失败或需要重新收敛时，再回到计划模式。
  7. 关键阶段完成后，再做一次独立复查。

说明：并行代理或辅助执行更适合复查、补充视角和并行审视，不适合承接必须立刻解决的主线任务。

最后讲“拿来即用”

这一层解决的是“拿来即用”。需要马上复制时，直接用模板和通用补充句。

器

四个推荐组合，建议先按这条顺序来用。

01

方案讨论型：先收敛关键分歧，再拍板

02

直接开发型：主线明确后，直接推进

03

Bug 修复型：先复现，再修复

04

独立复查型：主流程完成后，再独立审视

这四组都不是零散句子，而是把“目标、模式、约束、输出、验证”压成可直接复制的完整骨架。

器 · 组合 01

方案讨论型：先收敛关键分歧，再拍板。

目标：先收敛【某个方案 / 某个模块怎么做】，先不急着写代码。

模式：必要时先做计划收敛。

约束：把真正关键的分歧点提炼出来，不要把低价值问题一起抛给我。

输出：按“问题、选项、利弊、推荐”格式整理，我来拍板。

验证：明确这次准备解决什么、不解决什么，以及为什么。

器 · 组合 02

直接开发型：主线明确后，直接推进。

目标：解决【具体问题】，并尽量逼近【最终可用方案】。

模式：如果任务已经明确，就直接推进。

约束：先读上下文再动手；非关键问题不要频繁停下来；这次只解决【当前核心问题】，不顺手扩展到【无关范围】。

输出：先给问题判断和推荐方案，再直接落地；如果有必须我拍板的问题，按“问题、选项、利弊、推荐”来问我。

验证：改完后补充【测试 / 最小验证闭环】。

器 · 组合 03

Bug 修复型：先复现，再修复。

目标：修复【某个 bug】，并避免盲改。

模式：一般不需要先做计划收敛，除非背后牵涉较大范围的方案调整。

约束：优先先复现，再修复；不要为了让测试通过而绕开真实问题。

输出：说明 bug 根因、如何复现、代码如何修复。

验证：确认问题先被稳定复现，再确认修复后通过，并补充必要的回归验证。

器 · 组合 04

独立复查型：主流程完成后，再独立审视。

目标：对当前结果做一轮独立复查，而不是重复主流程。

模式：适合在主流程基本完成后使用。

约束：从【前端 / 架构 / 接口 / 文档】视角重点检查【稳定性 / 可维护性 / 一致性】；聚焦复查，不重复实现。

输出：指出风险、遗漏、更优建议，以及哪些点已经足够，不需要继续投入。

验证：把“必须改”和“可选优化”分开列出。

器

通用补充句

这次只解决当前核心问题，不顺手扩展、不做无关重构。

这次不考虑兼容过渡，直接按目标状态修改；不要额外保留兼容层、桥接逻辑或过渡方案。

如果有需要我确认的问题，请只提最关键的 1 到 3 个，并按“问题、选项、利弊、推荐”格式来问。

先列出你的关键假设和缺失信息；能自行拍板的直接继续，不能拍板的最后集中列出。

器

当你不希望它擅自用临时解时

适用

你更在意方案是否自然、易维护；你不接受它在未确认前直接使用兜底方案。

模板

如果过程中出现兜底方案，不要直接采用。先说明更优正解是什么、为什么暂时不可行；只有在确认没有更合适方案后，再向我说明风险、局限和回收方式，得到我确认后再实施。
